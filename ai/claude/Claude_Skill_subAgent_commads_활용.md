1. 서브에이전트 — 컨텍스트 관리가 핵심입니다
   AI와 오래 대화하다 보면 오토컴팩트(컨텍스트 요약)가 진행되면서, AI가 앞에서 한 얘기를 슬슬 까먹기 시작합니다. 이건 AI의 한계가 아니라 구조적인 문제입니다. 대화가 길어질수록 컨텍스트(맥락 정보)가 쌓이고, 그만큼 처리해야 할 정보가 많아지면서 정확도가 떨어지고 속도도 느려집니다.
   서브에이전트는 이 문제를 해결합니다.
   "코드 검토는 이 친구", "테스트는 저 친구" 식으로 업무별 담당자를 따로 두는 겁니다. 각 서브에이전트는 자기만의 독립된 컨텍스트에서 작업하고, 결과만 메인 에이전트에 보고합니다.
   이게 왜 중요하냐면

- 메인 대화창의 컨텍스트가 가벼워져서 AI가 핵심에 집중할 수 있습니다.
- 각 전문 영역에서 더 정확한 결과물이 나옵니다.
- 작업 속도가 체감될 정도로 빨라집니다

저는 요즘 필요한 서브 에이전트들을 직접 만들어서 활용을 하고 있습니다.
만드는 방법은 어렵지 않습니다.
그냥 클로드코드에게 요청을 해도 되고, 서브 에이전트를 만드는 오픈소스나 아래에서 소개해드리는 링크에서 스킬을 받아서 사용해보셔도 됩니다.
저는 이렇게 작업을 진행하다 보니 복잡한 프로젝트에서 메인 에이전트가 "멍해지는" 현상이 확 줄었습니다.

<기본 서브 에이전트 구현 방법>
.claude/agents/ 폴더에 마크다운 파일 생성. /agents명령어로 관리 가능.


2. 스킬 — 반복 설명 없이 전문성을 발휘하게 만드는 법
   매번 "발표자료 만들어줘. 슬라이드 10장 내외로, 목차 먼저 보여주고, 핵심 메시지는 굵게 처리하고…" 이렇게 설명하시나요?
   스킬은 이런 설명을 미리 저장해두는 겁니다. 한번 세팅해두면 Claude가 상황에 맞게 알아서 해당 스킬을 불러옵니다.
   아래에서 소개해 드릴 슬래시 커맨드랑 다른 점이 뭐냐면, 커맨드는 내가 직접 "/명령어"를 쳐야 실행되지만, 스킬은 Claude가 "아, 이건 저 스킬 쓰면 되겠다" 하고 자동으로 판단합니다.
   핵심은 description(설명)을 잘 쓰는 겁니다. Claude는 이 설명을 보고 언제 이 스킬을 쓸지 결정하거든요. "API 설계할 때 사용", "엑셀 파일 다룰 때 사용" 이런 식으로 사용 상황을 명확히 적어야 합니다.
   이런 스킬을 잘 만들어둔 곳이 skillsmp.com 입니다.
   커뮤니티가 만든 스킬이 37,000개 넘게 있습니다. 코드 리뷰, 문서 작성, 데이터 분석 등 실무에서 바로 쓸 수 있는 것들이 많습니다.
   직접 만들 필요 없이 여기서 가져다 쓰세요
   저희 ANYON 서비스도 여기서 프론트 디자인 스킬을 가져와서 쓰는데, 퀄리티가 좋아서 자주 사용중입니다.

<기본 스킬 구현 방법>
설정 방법: ~/.claude/skills/(개인용) 또는 .claude/skills/(프로젝트용) 폴더에 SKILL.md 파일 생성.


3. 슬래시 커맨드 — 하루에 수십 번 치는 반복을 없애는 법
   "코드 리뷰해줘. 보안 취약점 확인하고, 성능 이슈 체크하고, 코드 스타일도 확인해줘."
   이런 요청을 하루에 몇 번이나 치시나요? 저는 이걸 /review 한 단어로 줄였습니다.
   슬래시 커맨드는 자주 쓰는 프롬프트를 파일로 저장해두고 단축키처럼 쓰는 기능입니다. 단순해 보이지만 실제로 쓰면 작업 흐름이 달라집니다.
   제가 쓰는 커맨드 예시:
   /morning — git pull, 테스트 실행, 현재 상태 확인을 한 번에
   /ready — 커밋 전 린트, 타입 체크, 테스트 한 번에 돌리기
   /branch — 새 브랜치 생성하고 이동

<기본 슬래시커맨드 구현 방법>
설정 방법: .claude/commands/ 폴더에 마크다운 파일로 저장. 파일명이 곧 커맨드명입니다.

여기서 제가 추천드리는 오픈소스가 있습니다.
BMAD-METHOD인데요.
GitHub 스타 26,000개가 넘는 오픈소스 프로젝트인데, 저도 애용하고 있고, 저에게 맞게 커스터마이징을 하기 위해서 분석도 했는데, 계속 업그레이드가 되고있어서 활용하면 좋은 오픈소스입니다.
핵심 구성:
21개의 전문 AI 역할 (기획자, 설계자, 개발자, 디자이너, 테스트 설계자 등)
50개 이상의 워크플로우 템플릿
4단계 개발 사이클: 분석 → 기획 → 설계 → 실행
이 오픈소스가 좋은 점은 프로젝트 규모에 맞게 자동으로 조절된다는 겁니다:
간단한 버그 수정 → 기술 스펙만 작성하고 바로 시작
중간 규모 기능 → PRD + 설계 + UX 검토 후 진행
대규모 프로젝트 → 전체 거버넌스 프로세스 적용
설치 후 /workflow-init 한 번 실행하면 내 프로젝트 상황을 분석해서 적절한 트랙을 추천해줍니다.
링크: https://lnkd.in/ghpVk5CZ
다음에 기회가 된다면 실제로 써먹으면 좋을 워크플로우를 소개하겠습니다.


정리하면
서브에이전트 = 컨텍스트 분리로 정확도와 속도 확보
스킬 = 상황 맞춤 전문성 자동 적용
슬래시 커맨드 = 반복 프롬프트 완전 자동화
세 가지 모두 한번 세팅해두면 그 다음부터는 계속 씁니다.
투자 대비 효과가 확실한 영역이에요.

---

plan은 phase 별로 나누어서 세운다. 

특정 디렉토리에 plan을 저장하여 관리하고 하나식 phase를 구분해서 구현한다.

이렇게 하면 Phase 구현 중간에 작업이 끊겨도 plan.md으로부터 구현해야 하는 phase의 컨텍스트를 다시 불러와서 동일한 작업을 쉽게 이어갈 수 있다.

---

💡 먼저 결론부터
실행을 보장하려면 Hook, 사고를 고정하려면 Skill, 상황을 읽어 적용하려면 Agent.
셋 다 “Claude가 더 잘 일하게 만드는 장치”인데, 담당하는 영역이 다릅니다.

▪️ Skill: 각 단계에서 “어떤 기준으로” 할지 정의
▪️ Hook: 단계 사이에서 “빠지면 안 되는 검사”를 강제 실행
▪️ Agent: 단계별로 “누가” 담당할지 역할을 분리

이렇게만 보면 추상적이니까, 실제로 겪었던 상황들로 풀어볼게요.

—
📋 상황 1: 매번 같은 말을 반복하고 있다
“함수는 짧게 쪼개고, 변수명은 명확하게, 주석은 왜(why)를 설명해.”
코드를 요청할 때마다 이 말을 붙여야 했습니다. 안 붙이면 결과물이 들쑥날쑥했거든요. 어떤 날은 함수가 100줄이고, 어떤 날은 주석이 하나도 없고.
이건 Skill로 해결했습니다. 한 번 적어두면 매번 말 안 해도 Claude가 그 기준으로 작성합니다. Skill은 “이렇게 생각해”를 고정하는 장치입니다.

—
🔧 상황 2: 분명히 적어뒀는데 안 지켜진다
Skill에 “코드 작성 후 린트 검사 돌려”라고 적어뒀습니다. 그런데 가끔 안 돌아갔습니다. Claude가 까먹은 건지, 맥락상 생략한 건지 모르겠지만 어쨌든 누락이 생겼습니다.
이건 Hook으로 옮겼습니다. Hook은 Claude가 기억하는 게 아니라 시스템이 강제로 실행합니다. 코드가 저장되면 자동으로 린트가 돌아갑니다. 사람이 깜빡해도, Claude가 건너뛰어도, 훅은 그냥 실행됩니다.

Skill은 지침이고, Hook은 게이트입니다. “하면 좋은 것”은 Skill에, “안 하면 안 되는 것”은 Hook에 두는 게 맞더라고요.

—
🤖 상황 3: 대화가 길어지니까 앞뒤가 안 맞는다
코드를 짜달라고 하고, 이어서 리뷰해달라고 하고, 그다음 문서화해달라고 했습니다. 그런데 대화가 길어지니까 이상해지더라고요. 리뷰하면서 자기가 짠 코드를 감싸거나, 문서 쓰면서 아까 리뷰에서 지적한 걸 까먹거나.

이건 Agent로 분리했습니다. 구현하는 Claude, 리뷰하는 Claude, 문서 쓰는 Claude를 따로 뒀습니다. 리뷰 Agent는 코드를 짠 맥락을 모르니까 오히려 객관적으로 봅니다. 역할이 섞이지 않으니까 결과물이 깔끔해졌습니다.

다만 처음에 너무 잘게 쪼갰다가 관리가 복잡해진 적도 있습니다. “기능”으로 나누는 게 아니라 “역할”로 나눠야 한다는 걸 뒤늦게 알았어요. 리뷰어, 테스터, 문서 작성자처럼 산출물이 명확한 단위로요. 개인 프로젝트라면 3개 내외가 적당한 것 같습니다.

—
🎯 정리하면
같은 지시를 매번 반복한다 → Skill (사고 기준 고정)
적어뒀는데 누락된다 → Hook (실행 강제)
대화가 길어지면 산만해진다 → Agent (역할 분리)

반대로 말하면 이렇습니다.

Skill에만 두면 누락 위험이 있고,
Hook으로 다 만들면 느려지고,
Agent를 과하게 쪼개면 복잡해집니다.

적절한 자리에 두는 게 포인트입니다.

—
💭 쓰면서 느낀 것
결국 이 세 개념은 “Claude를 어떻게 통제할 것인가”의 문제인 것 같습니다.

Skill은 생각의 방향을 잡아줍니다. 하지만 강제는 아닙니다.
Hook은 실행을 보장합니다. 하지만 판단은 못 합니다.
Agent는 역할을 나눕니다. 하지만 너무 나누면 흩어집니다.

정리하고 나니 조금 감이 잡히는 것 같기도 합니다. 다만 실제로 새로운 기능을 만들 때 “이건 어디에 둬야 하지?”라는 판단은 여전히 어렵습니다. 그래서 아까 말한 것처럼 분류 전담 에이전트를 둘까 싶다가도, 그건 또 배보다 배꼽인 것 같기도 하고요.
