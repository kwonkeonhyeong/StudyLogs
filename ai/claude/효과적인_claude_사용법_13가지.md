코드팩토리 영상에서 소개된, 클로드 코드(Claude Code)의 창시자 보리스 체니(Boris Chen)가 직접 사용하는 13가지 활용 팁을 정리해 드립니다. 보리스는 이 방법들을 통해 혼자서 일주일에 PR 100개를 처리한다고 합니다.

**1. 로컬 병렬 실행 (Parallel Execution)**
보리스는 기본적으로 로컬에서 5개의 클로드 세션을 동시에 실행합니다. 사용자 입력이 필요할 때 시스템 알림이 뜨도록 설정하여 클로드의 요청에 즉시 반응할 수 있게 합니다.

- CLI의 경우에는 터미널에서 실행하는 claude마다 독립적인 세션을 가진다. 웹의 경우 대화창마다 독립적인 세션을 가진다. 각 세션은 컨텍스트가 독립적으로 관리되며 대화가 공유되지 않는다.
- 사용량에 측정되는 Session 사용량은 전체 세션 사용량을 의미한다.
- 병렬 작업을 진행할 때, 같은 소스를 수정할 수 있는 작업을 병렬로 진행하면 충돌 위험이 있기 때문에, 서로 영향을 끼치지 않는 작업들을 병렬로 실행한다.

**2. 웹 세션 및 백그라운드 활용**
로컬 세션 외에도 5~10개의 웹 세션을 추가로 실행하여 총 15개 정도의 세션을 동시에 활용합니다. 또한, 명령어 뒤에 `&`를 붙여 작업을 즉시 백그라운드 태스크로 넘기는 기능을 적극적으로 사용합니다.

- 오래 걸리는 작업의 경우에는 백그라운드 태스크로 넘겨서 진행하는 것이 효율적이라고 생각함.

**3. 똑똑한 모델 사용 (Opus 4.5 & Thinking Mode)**
작은 모델보다는 항상 오퍼스(Opus) 4.5를 사용하며, 씽킹(Thinking) 모드를 켜둡니다. 모델이 무거워도 초기에 제대로 계획을 세우고 사람의 개입을 줄이는 것이 결과적으로 작업 속도를 훨씬 빠르게 만든다고 합니다.

- /model 을 통해서 이용하는 모델을 변경할 수 있다. 
- claude는 기본적으로 Thinking 모드로 설정되어 있기 때문에 따로 켤 필요는 없다.

**4. CLAUDE.md 파일의 팀 공유**
`CLAUDE.md`는 깃(Git)에 커밋하도록 설계된 파일입니다. 보리스는 클로드가 실수를 하면 다음에는 같은 실수를 하지 않도록 지침(Instruction)을 이 파일에 추가하여 저장하며, 팀원들과 적극적으로 파일을 개선하고 공유합니다.

- LLM은 관련된 집중적인 컨텍스트로 가득 찬 컨텍스트 윈도우가 있을 때, 더 나은 성능을 보입니다.
- 컨텍스트 윈도우의 경우

- [시스템 프롬프트] + [대화 히스토리] + [현재 입력] + [생성 중인 응답]
  └────────────────── 모두 합쳐서 컨텍스트 윈도우 ──────────────────┘

- 위와 같은 형태를 가지는데, 중요한 점은 대화가 길어질수록 컨텍스트가 쌓이며 한계에 도달하면 오래된 대화부터 잘리거나 (sliding window) 또는 새로운 세션 시작을 필요로 하게 된다.
- 그렇기 때문에 매 세션마다 적용되는 CLAUDE.md가 너무 많은 토큰을 차지하게 되면 컨텍스트 윈도우가 낭비될 수 있다.

**5. PR 검증과 GitHub 연동**
PR(Pull Request)을 검증할 때 클로드를 태그하여 `CLAUDE.md`에 내용을 추가하게 합니다. 또한, 인스톨 커맨드를 사용하여 깃허브에서도 클로드를 바로 사용할 수 있도록 설정합니다.

**6. 플랜 모드(Plan Mode)로 시작**
작업을 시작할 때 시프트+탭(Shift+Tab)을 눌러 플랜 모드로 진입합니다. 계획을 충분히 수정한 뒤 '오토 액셉트(Auto Accept) REPL 모드'로 실행하면, 보통 한 번에 원하는 결과물을 얻을 수 있다고 합니다.

**7. 커스텀 커맨드 활용**
반복되는 워크플로우나 프롬프트는 커맨드로 저장하여 재사용합니다. 이를 통해 배시(Bash) 커맨드 사용 권한도 줄 수 있어 입력을 크게 줄일 수 있으며, 이 커맨드들은 깃에 커밋하여 팀원과 공유합니다.

**8. 서브 에이전트(Sub-agents) 사용**
특정 목적을 위한 서브 에이전트를 규칙적으로 사용합니다. 예를 들어 코드 정리를 위한 '코드 심플리파이어(Code Simplifier)'나 엔드투엔드 테스트 명세가 적힌 '베리파이(Verify)' 에이전트 등을 통해 작업을 자동화합니다.

**9. 포스트 툴 훅(Post-tool Hooks)으로 포매팅**
클로드가 코드를 작성한 후 자동으로 포매팅이 되도록 훅을 사용합니다. 이는 혹시 모를 10%의 사소한 문제를 자동으로 해결해 줍니다.

**10. 권한 관리 (Permission)**
'dangerously skip permission' 모드를 사용하는 대신, 자주 쓰는 배시 커맨드들을 미리 등록해 두고 지속적인 권한 허가가 필요 없도록 설정합니다. 이 권한 설정도 `settings.json`에 저장해 팀과 공유합니다.

**11. 광범위한 툴 사용 권한 제공**
클로드에게 슬랙(Slack) 검색, 빅쿼리(BigQuery) 실행, 센트리(Sentry) 에러 로그 확인 등 다양한 툴을 사용할 수 있는 권한을 모두 제공하여 활용도를 높입니다.

**12. 장기 실행 작업(Long-running Tasks) 핸들링**
오래 걸리는 작업은 백그라운드 에이전트로 검증하게 하거나, 에이전트 스톱 훅을 사용해 결정적인(deterministic) 결과를 얻습니다. 또한 'Will Gone' 플러그인 등을 사용해 무한 반복 작업을 수행하기도 합니다.

**13. 검증 방법의 구체적 제시**
클로드에게 자신이 한 작업을 검증할 수 있는 정확한 방법(브라우저 테스트, 모바일 시뮬레이션, 배시 커맨드 등)을 제공합니다. 클로드 스스로 피드백을 생성하고 검증하게 하면 결과물의 품질이 2~3배 이상 좋아진다고 합니다.

이 13가지 팁은 단순한 코딩 팁이 아니라, 개발자가 갖춰야 할 '필수 생존 키트'와 같다고 영상에서는 강조하고 있습니다.
